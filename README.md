# Система бронирования ресурсов

## Описание проекта

Система бронирования ресурсов (переговорные комнаты, оборудование) с разграничением прав пользователей и администраторов. Проект включает клиентскую часть (React), серверную часть (Node.js + Express), реляционную базу данных PostgreSQL, Redis для кэширования и синхронизации, MongoDB для хранения логов и аналитики.

## Функциональные возможности

### Для пользователей
- Регистрация и аутентификация с использованием JWT.
- Защита от перебора паролей: после трёх неудачных попыток вход блокируется на 10 минут (Redis).
- Просмотр списка доступных ресурсов.
- Создание и отмена собственных бронирований.

### Для администраторов
- Все возможности пользователя.
- Управление пользователями: создание, удаление, смена роли (пользователь/администратор), блокировка/разблокировка.
- Управление ресурсами: создание, редактирование, удаление.
- Просмотр логов системы (хранятся в MongoDB) с фильтрацией по пользователю, действию, уровню, интервалу дат, экспорт в CSV.
- Аналитические отчёты на основе логов MongoDB:
  - Статистика активности пользователей по периодам (день/неделя/месяц).
  - Топ-10 самых активных пользователей.
  - Распределение операций по типам (CRUD-статистика).
  - Временные тренды (почасовая/подённая динамика).
  - Выявление аномалий (пользователи, совершившие более заданного количества событий за час).

### Технические механизмы
- Кэширование часто запрашиваемых данных (списки пользователей, ресурсов, справочники) в Redis с настраиваемым TTL.
- Инвалидация кэша при операциях INSERT/UPDATE/DELETE с использованием Redis Pub/Sub для синхронизации между несколькими инстансами приложения.
- Хранение сессий пользователей в Redis (чёрный список JWT-токенов).
- Логирование всех действий пользователей и системных ошибок в MongoDB с автоматическим удалением старых записей (TTL-индекс).

## Технологический стек

- **Frontend**: React, React Router, Axios.
- **Backend**: Node.js, Express.
- **Реляционная БД**: PostgreSQL.
- **Кэш и pub/sub**: Redis (ioredis).
- **Документная БД**: MongoDB (Mongoose).
- **Аутентификация**: JWT, bcrypt.
- **Экспорт отчётов**: json2csv.

## Установка и настройка

### Требования
- Node.js (версия 18 или выше).
- PostgreSQL (13 или выше).
- Redis (6 или выше) – может быть локальным или облачным.
- MongoDB (5 или выше) – локальным или облачным (Atlas).

### Шаги по установке

1. Клонируйте репозиторий:
   ```bash
   git clone <url-репозитория>
   cd resource-booking-system
   ```

2. Установите зависимости для сервера:
   ```bash
   npm install
   ```

3. Установите зависимости для клиента:
   ```bash
   cd client
   npm install
   cd ..
   ```

4. Создайте файл `.env` в корневой директории со следующим содержимым (замените значения на свои):
   ```env
   # PostgreSQL
   DB_HOST=localhost
   DB_PORT=5432
   DB_USER=postgres
   DB_PASSWORD=your_password
   DB_NAME=resource_booking

   # Redis
   REDIS_HOST=your_redis_host
   REDIS_PORT=your_redis_port
   REDIS_PASSWORD=your_redis_password

   # MongoDB
   MONGODB_URI=mongodb+srv://user:password@cluster.mongodb.net/resource_booking

   # JWT
   JWT_SECRET=your_secret_key

   # Порт сервера (опционально)
   PORT=3000
   ```

5. Инициализируйте базу данных PostgreSQL (создайте таблицы users, resources, bookings и т.д.). Скрипты находятся в папке `database/`.

6. Запустите сервер:
   ```bash
   npm run dev
   ```
   Сервер будет доступен по адресу `http://localhost:3000`.

7. Запустите клиентское приложение:
   ```bash
   cd client
   npm start
   ```
   Клиент будет доступен по адресу `http://localhost:3001`.

## Руководство пользователя

### Регистрация и вход
- Перейдите на страницу `/register` для создания нового аккаунта. Укажите логин, пароль, email, имя и фамилию.
- Для входа используйте страницу `/login`. После трёх неудачных попыток входа под одним логином доступ блокируется на 10 минут, о чём выводится сообщение с таймером.

### Главная страница
После входа отображается панель управления (dashboard) с карточками:
- **Ресурсы** – список доступных ресурсов.
- **Мои бронирования** – список ваших броней с возможностью отмены.
- **Администрирование** (видна только администраторам) – переход к управлению пользователями, ресурсами, логами и аналитике.

### Бронирование ресурса
1. В разделе «Ресурсы» выберите нужный ресурс.
2. Укажите дату и время начала/окончания бронирования.
3. Подтвердите создание бронирования. При успехе вы увидите уведомление.

## Руководство администратора

### Управление пользователями
- Перейдите в раздел «Администрирование» → «Пользователи».
- **Создание пользователя**: заполните форму (логин, пароль, email, роль).
- **Удаление**: пользователь удаляется вместе со всеми связанными бронированиями (транзакция).
- **Смена роли**: переключение между обычным пользователем и администратором.
- **Блокировка/разблокировка**: заблокированный пользователь не может войти в систему.
- После любого изменения кэш списка пользователей инвалидируется (удаляется) во всех инстансах приложения через Redis Pub/Sub.

### Управление ресурсами
- **Создание ресурса**: укажите название, тип, вместимость, локацию (здание).
- **Редактирование**: измените любые поля.
- **Удаление**: ресурс можно удалить только если на него нет активных бронирований.

### Просмотр логов
- Раздел «Логи» (MongoDB). Доступны фильтры:
  - ID пользователя.
  - Тип действия (LOGIN, CREATE_USER, UPDATE_RESOURCE и т.д.).
  - Уровень (info, error).
  - Интервал дат (начало и конец).
- Результаты отображаются в таблице с пагинацией. Доступна сортировка по дате.
- Кнопка «Экспорт CSV» позволяет выгрузить текущий отфильтрованный список в CSV-файл.

### Аналитические отчёты
- Раздел «Аналитика». Доступны вкладки:
  - **Активность по периодам**: выбор периода (день/неделя/месяц), отображение количества событий и уникальных пользователей.
  - **Топ пользователей**: лимит (по умолчанию 10), отображение ID пользователя, числа событий, последнего события и списка действий.
  - **Распределение действий**: частота каждого типа действия.
  - **Временные тренды**: интервал (час/день/неделя), таблица с временными слотами и детализацией по действиям.
  - **Аномалии**: порог (по умолчанию 100 событий за час). Вывод пользователей, превысивших порог, с указанием периода и примеров событий.
- Все отчёты можно экспортировать в CSV.

### Примечания
- Административные разделы доступны только пользователям с ролью admin.
- Логи в MongoDB автоматически удаляются через 30 дней (TTL-индекс).
- Для горизонтального масштабирования достаточно запустить несколько экземпляров приложения, подключённых к общим Redis и MongoDB. Синхронизация кэша обеспечивается через Pub/Sub.
- Пароли пользователей хранятся в PostgreSQL в захешированном виде (bcrypt).
